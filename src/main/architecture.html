OneLog - Desktop Application
Version 1.0.0

1. System Overview

OneLog is a lightweight, offline-first desktop application for working professionals to log daily activities, track tasks with time, manage projects, and plan ahead. All data is stored locally - nothing leaves the user's machine.

3. Architecture Details

Click any section below to expand.

3.1 Main Process Electron

The main process (src/main/main.js) is the entry point of the application. It runs in a Node.js environment with full OS access.

Responsibilities:

  • Window Management - Creates the main BrowserWindow (1000×700, min 800×550) and secondary help/architecture windows
  • Application Menu - Builds a native menu bar with File, Tools (Import/Export), Edit, View, and Help menus
  • IPC Handler Registration - Registers 40+ ipcMain.handle() channels for database operations
  • Database Initialization - Calls db.initialize() and db.runSeedIfNeeded() on startup
  • File Dialogs - Native open/save dialogs for import/export via dialog.showOpenDialog() / dialog.showSaveDialog()
  • CSV Generation - Server-side CSV formatting with proper escaping for export

Lifecycle:

  1. app.whenReady() → Initialize DB → Register IPC → Build Menu → Create Window
  2. In dev: loads http://localhost:5174 (Vite dev server)
  3. In prod: loads dist/renderer/index.html from asar archive
3.2 Preload & IPC Bridge Security

The preload script (src/main/preload.js) runs in a sandboxed context before the renderer page loads. It bridges main ↔ renderer communication securely.

Security model:

  • contextIsolation: true - Renderer cannot access Node.js APIs directly
  • nodeIntegration: false - No require() in renderer
  • contextBridge.exposeInMainWorld('api', {...}) - Exposes a curated, typed API surface as window.api

Communication patterns:

  • Request/Response: ipcRenderer.invoke(channel, args) → returns a Promise (used for all CRUD operations)
  • Menu Events: ipcRenderer.on(channel, callback) → listens for menu-triggered actions (export, import-complete). Returns a cleanup function for React useEffect.

State management: All state lives in App.tsx using React hooks (useState, useEffect, useCallback). No external state library - the app's data complexity doesn't warrant one.

Key patterns:

  • Auto-save: Debounced at 800ms via useEffect + setTimeout
  • Timer auto-stop: Starting a timer stops all other running timers. Completing a task auto-stops its timer.
  • Date handling: Shared formatDate() utility in src/renderer/utils.ts uses local date parts (not toISOString()) to avoid timezone shifts
  • Menu integration: useEffect hooks subscribe to onMenuExport / onMenuImportComplete events with proper cleanup
3.4 Rich Text Editor Tiptap

Both the Daily Log and Tomorrow's Plan use the RichTextEditor component, built on Tiptap (a headless ProseMirror wrapper).

Extensions loaded:

  • StarterKit - paragraphs, headings (H2, H3), bold, italic, strike, lists, blockquotes, code blocks, horizontal rules
  • Underline - underline formatting
  • Placeholder - grey placeholder text when empty
  • ResizableImage - custom Node extension for inserting and resizing images inline (drag-to-resize handles)
  • Backlink - custom inline atom Node extension for date cross-references. Input rule: [[YYYY-MM-DD]] auto-converts to a clickable blue date chip. HTML serialization: <span data-type="backlink" data-date="2026-01-15"></span>

Formatting toolbar: Bold, Italic, Underline, Strike, H2, H3, Bullet List, Ordered List, Blockquote, Code Block, Horizontal Rule, Insert Image

Data format: Content is stored as HTML strings in the SQLite database. Tiptap serializes/deserializes to/from HTML via editor.getHTML() and the content prop. Backlink nodes are stored as <span data-type="backlink" data-date="..."> elements, enabling database search for cross-references.

4.3 Database Configuration & Pragmas
  • Engine: better-sqlite3 - synchronous, in-process SQLite (no async overhead)
  • File location: %APPDATA%\mydailydiary\mydailydiary.db
  • Journal mode: WAL (Write-Ahead Logging) - better concurrent read performance and crash resilience
  • Foreign keys: ON - enforced at the database level
  • Indexes: idx_tasks_date on tasks(entry_date), idx_entries_date on entries(date), unique constraint on entry_tags(entry_date, tag_id)

Migration strategy: Additive ALTER TABLE wrapped in try/catch - checks if column exists before adding. No versioned migration files needed for this scale.

4.4 Key Query Patterns

Task retrieval with project JOIN:

SELECT t.*, p.name as project_name, p.color as project_color
FROM tasks t
LEFT JOIN projects p ON t.project_id = p.id
WHERE t.id = ?

This pattern is extracted into a getTaskById(id) helper used by all task mutation functions (add, toggle, update, timer operations).

Entry upsert (insert or update):

INSERT INTO entries (date, log_content, tomorrows_plan)
VALUES (?, ?, ?)
ON CONFLICT(date) DO UPDATE SET
  log_content = excluded.log_content,
  tomorrows_plan = excluded.tomorrows_plan,
  updated_at = datetime('now')

Backlink search (cross-references):

SELECT date, log_content, tomorrows_plan FROM entries
WHERE (log_content LIKE '%data-date="2026-01-15"%'
   OR tomorrows_plan LIKE '%data-date="2026-01-15"%')
  AND date != '2026-01-15'
ORDER BY date DESC LIMIT 50

Searches stored HTML for backlink node markers (data-date attributes) to find all entries that reference a given date.

Workload heatmap aggregation:

SELECT entry_date as date, COUNT(*) as task_count,
  SUM(CASE WHEN completed=1 THEN 1 ELSE 0 END) as completed_count,
  SUM(COALESCE(time_spent,0)) as total_time
FROM tasks WHERE entry_date >= ? AND entry_date <= ?
GROUP BY entry_date

Import transaction: Uses db.transaction() to wrap project, entry, and task inserts in a single atomic operation. Projects are imported first so tasks can resolve project_name → project_id.

  │

  │

8. Build & Deployment

8.1 Development Workflow 
npm run dev

Runs two processes concurrently:

  1. Vite dev server (npm run dev:renderer) - starts on localhost:5174 with HMR
  2. Electron (npm run dev:electron) - waits for Vite to be ready, then launches electron .

The wait script (scripts/wait-for-vite.js) polls localhost:5174 until it responds before launching Electron to prevent a blank window.

8.2 Production Build 
npm run build

Executes two stages:

  1. vite build - Bundles React/TS/Tailwind into dist/renderer/ (index.html + hashed JS/CSS assets)
  2. electron-builder --win - Packages into a Windows installer:
    • Compiles better-sqlite3 native module for Electron's Node.js version
    • Packs src/main/**/* + dist/renderer/**/* into an asar archive
    • Generates NSIS installer (~97 MB) in release/

Installer options (NSIS):

  • Per-user install (no admin required)
  • Custom install directory
  • Desktop + Start Menu shortcuts
  • App ID: com.iqzsystems.dailydiary
8.3 Native Module Handling

better-sqlite3 is a C++ native addon that must be compiled for the correct Node.js ABI version.

⚠️ The native module must be compiled for the target runtime:
• System Node.js - for running tests (npm rebuild better-sqlite3)
• Electron's Node.js - for running/packaging the app (npx electron-rebuild)

The pretest npm script automatically runs npm rebuild better-sqlite3 before tests to ensure the native module matches the system Node.js version.

electron-builder runs @electron/rebuild automatically during the build step.

11. Security Considerations

11.1 Electron Security Model
  • Context Isolation: Enabled - renderer runs in a separate context from preload
  • Node Integration: Disabled - no require() or Node APIs in renderer
  • Context Bridge: Only whitelisted API methods are exposed via window.api
  • No Remote Module: The deprecated remote module is not used
  • Offline-only: No network requests, no external API calls, no telemetry
  • DevTools: Only available in dev mode (not exposed in production builds)
11.2 Data Security
  • Local storage only: SQLite database stored in user's AppData - no cloud sync
  • No encryption at rest: Data is stored in plaintext SQLite. Relies on OS-level user account protection. Suitable for personal productivity data.
  • Parameterized queries: All SQL uses prepared statements with ? placeholders - no string concatenation, preventing SQL injection
  • CSV export escaping: Values containing commas, quotes, or newlines are properly quoted with double-quote escaping
  • Import validation: Input is validated (type checks, required fields) before database insertion

12. First-Launch Experience

Seed Data & Guided Tour

On first launch (no onboarding_complete setting exists):

  1. db.runSeedIfNeeded() calls seedSampleData(db) which inserts:
    • 5 projects (ETL Pipeline, Data Quality, Reporting, Infrastructure, Meetings)
    • 8 tags (pipeline, spark, azure, incident, documentation, optimization, sprint, demo)
    • 6 holidays (New Year, MLK Day, Republic Day, Valentine's Day, etc.)
    • 4 task templates (Daily Standup as recurring, Sprint Planning, Data Pipeline Deploy, Month-End Close)
    • 27 working days of entries (Jan 2 – Mar 3, 2026) with realistic data engineer diary content
    • ~150 tasks with completion status, time tracking, project assignments, and Kanban statuses (todo/in_progress/done)
    • 8 backlink cross-references between related entries (e.g., incident follow-ups, sprint reviews, demo references)
    • Tag associations across all entries for filtering and categorization
  2. Sets onboarding_complete = 'pending_tour'
  3. Renderer detects pending_tour → shows the GuidedTour overlay
  4. Tour walks through 9 steps: sidebar, search, daily log, tasks, timer, projects, carry-over, export (Tools menu), tomorrow's plan
  5. On tour completion, sets onboarding_complete = 'true'

Subsequent launches: runSeedIfNeeded() finds the setting exists and skips seeding.

After resetting data: the setting is cleared, so the next launch re-seeds and re-shows the tour.

13. Performance Characteristics

Runtime Performance
  • SQLite (synchronous): All database operations are synchronous via better-sqlite3, avoiding async overhead. Typical query: <1ms for single-row operations.
  • Auto-save debounce: 800ms delay prevents excessive writes during rapid typing. Only one saveEntry call per typing pause.
  • Timer updates: 1-second setInterval recalculates elapsed display client-side - no database writes during active timing (only on start/stop).
  • Search: Uses LIKE '%query%' with a 50-result limit. Fast enough for personal diary volumes (hundreds to low thousands of entries).
  • Renderer bundle: ~597 KB JS (190 KB gzipped). Single-chunk build - adequate for a desktop app loaded from local filesystem.
  • Memory: Typical footprint ~120–180 MB (Electron baseline + renderer + SQLite). No memory leaks from timer intervals (cleaned up via useEffect return).
